# Etapa 2 — Sub-phase plan

> **For agentic workers:** This plan is broken into 7 sub-phases. Each is committed and tested before moving to the next. Refer to `docs/superpowers/specs/2026-05-23-etapa-2-design.md` for the full design context. Progress is tracked in `docs/superpowers/etapa-2-progress.md`.

**Goal:** Implement etapa 2 in 7 user-approval-gated sub-phases. Each sub-phase has a branch, acceptance criteria, and a manual smoke list that the user runs before approving the next sub-phase.

**Spec:** `docs/superpowers/specs/2026-05-23-etapa-2-design.md`

## Workflow per sub-phase

1. Create branch `feat/etapa-2-N-<slug>` from `master` (or from previous sub-phase if dependent).
2. Implement tasks.
3. Run automated tests (`cargo test`, `pnpm test`).
4. Update `docs/superpowers/etapa-2-progress.md` marking the sub-phase as Ready for Review.
5. User runs the sub-phase's smoke list, types `aprovado` or requests changes.
6. On approval: merge to `master` (or tag), update progress to Done, start next sub-phase.

## Branching

- `master` always reflects the latest approved state.
- Each sub-phase has its own branch.
- Sub-phases 2.1+ generally depend on 2.0; later sub-phases depend on 2.1 (tokens + primitives).
- Don't squash — preserve granular commits for the progress trail.

## Sub-phase 2.0 — Close MVP gaps + cache

**Branch:** `feat/etapa-2-0-mvp-close`

**Scope:**

1. **Line-click comment** via Monaco view zones. Hover gutter shows `+`; click opens an inline draft editor (React mounted into a Monaco view zone via createRoot). Draft saved → renders inline as a thread widget on the same line.
2. **Multi-line range comment**. Capture editor selection (`getSelection().getStartPosition()` / `EndPosition`). A floating button overlays the selection; click creates a draft with `start_line=…, line=…, side=RIGHT`. Submit sends `start_line`/`start_side` to GitHub.
3. **Threads inline widget**. Existing review threads render as view zones too, anchored to their `line`. Reply textarea inside the widget. Resolve button calls `/pulls/N/comments/{id}` thread resolve (deferred — only ship reply for now).
4. **Reply on existing thread**. Backend already exposes `reply_to_thread`; surface it in the inline thread widget.
5. **SQLite cache active**.
   - `get_pr_diff`, `get_pr_threads`, `get_pr` all go through `cache::with_conn` first. TTL: 60 s for list endpoints (`list_prs` cached too), 5 min for PR detail / diff / threads.
   - `refresh_pr` deletes the cached rows for the PR before fetching.
   - Cache hit / miss logged at debug level.
6. **AppImage bundle**: retry `pnpm tauri build` after pre-downloading the linuxdeploy binaries manually (Tauri docs explain the env var); on failure, document the workaround and skip.
7. **Smoke run**: tick the etapa-1 manual checklist end-to-end.

**Acceptance smoke list:**

- [ ] Open a PR. Hover a line in the modified diff → `+` appears in the gutter.
- [ ] Click the `+` → inline editor opens at that line. Type, save. Draft renders as a widget on the same line. Reload PR → draft survives.
- [ ] Drag-select 3 lines → floating "Comentar 3 linhas" appears. Click → draft with start_line + line set.
- [ ] Existing review threads render inline as widgets at their lines. Reply text → posts as `in_reply_to`.
- [ ] Open same PR again — second open is markedly faster (cache hit visible in logs).
- [ ] Refresh PR button forces a fresh fetch (cache miss in logs).
- [ ] Submit review with the new drafts. Confirm GitHub web sees the inline + range comments correctly.
- [ ] AppImage exists OR build doc updated stating why it's deferred.

## Sub-phase 2.1 — Design system foundation

**Branch:** `feat/etapa-2-1-design-system`

**Scope:**

1. **Design tokens** in `src/styles/tokens.css`: spacing, radius, font sizes, weights, line heights, shadows, transitions. Imported by `globals.css`.
2. **Refactor `globals.css`** to consume tokens; CSS variables (`--space-2`, `--text-sm`, etc.).
3. **`src/components/ui/` primitives**:
   - `Button.tsx` (variants × sizes, focus ring, hover)
   - `Input.tsx`, `Textarea.tsx`
   - `Modal.tsx` (refactor of existing)
   - `Dropdown.tsx` / `Combobox.tsx`
   - `Tabs.tsx`
   - `Tooltip.tsx`
   - `Badge.tsx`
   - `Avatar.tsx`
   - `Spinner.tsx`
   - `EmptyState.tsx`
4. **Refactor existing components to consume primitives**. Visual behaviour should stay roughly identical (small polish OK). Heavy layout redesigns wait for 2.2+.
5. **Storybook-lite**: a hidden route `/__ui` that renders all primitives — useful for sanity check (optional but recommended).

**Acceptance smoke list:**

- [ ] App opens and behaves identically. Same flows.
- [ ] Buttons across the app share the same hover / focus / pressed states.
- [ ] All inputs (PAT, repo add, settings) share the same focus ring + padding.
- [ ] Modal padding and corner radius are consistent.
- [ ] Hovering rows in PR list and File tree feels uniform.
- [ ] No layout regressions — diff still renders, settings open, PR list populates.

## Sub-phase 2.2 — PR list redesign

**Branch:** `feat/etapa-2-2-pr-list`

**Scope:**

1. CI status indicator (🟢🟡🔴⚪) computed from `combined_status` or `check-runs` API. Add to `PrSummary`.
2. Search input persistent at the top, focuses on `Ctrl+P`. Filter applies on title / author / number client-side.
3. Tab counts (`Pra revisar (3)` / `Meus (1)`).
4. Repo group header shows repo name only; tooltip = `owner/repo`.
5. PR title truncate (1 line + ellipsis); tooltip = full title.
6. Meta line refactor: `author · age · N changed`.
7. Loading + empty states with `EmptyState` primitive.

**Acceptance smoke list:**

- [ ] PR list shows CI dot per PR.
- [ ] `Ctrl+P` focuses search; typing filters list.
- [ ] Tab counts reflect actual length.
- [ ] Hovering repo group header shows full `owner/repo` tooltip.
- [ ] Long PR titles show ellipsis; hovering shows full title.
- [ ] Empty state appears when no repos / no PRs (correct copy + CTA).

## Sub-phase 2.3 — Layout global (header, autocollapse, drawer)

**Branch:** `feat/etapa-2-3-layout`

**Scope:**

1. **Global header** above the panel group: breadcrumb `← repo › #N title`, `[Revisar]` action, settings gear. Replaces the bare gear bar from etapa 1.
2. **PR list autocollapse**: when `openPr` succeeds, collapse the PR list panel. Re-expand via toggle button or `Ctrl+1`.
3. **File tree → overlay drawer**: remove the file tree panel, render as a fixed-position overlay anchored to the left. Backdrop semi-transparent; `Ctrl+2` toggles; ESC and click-outside close.
4. **PR meta strip** under the header: `author · age · N files · +X −Y · CI · N reviewers`.
5. **PR description (body)** rendered collapsed; click to expand. Markdown → plaintext for now (full md render lives in 2.4 or later).

**Acceptance smoke list:**

- [ ] Open a PR — PR list shrinks to the left bar (icon only), diff fills the gap.
- [ ] Breadcrumb shows repo + number + title; clicking `←` returns to PR list (re-expands).
- [ ] `Ctrl+2` opens file tree overlay over the diff. ESC closes. Click outside closes.
- [ ] PR meta strip shows correct counts and CI badge.
- [ ] Description toggle works.

## Sub-phase 2.4 — Diff & comments redesign

**Branch:** `feat/etapa-2-4-diff-comments`

**Scope:**

1. **Replace ReviewSubmitModal with slide-in panel** (drafts panel) anchored to right edge. Same controls.
2. **"Mark viewed"** toggle per file. Persisted in a new SQLite `viewed_files (pr_id, path)` table. File header shows `viewed` chip.
3. **Diff render-mode setting**. New section in Settings → Diff. `Side-by-side | Inline | Auto`. Auto switches to inline below 1100 px window width. Toggle removed from diff header.
4. **Inline thread widget polish** — replies, resolve button (resolve via GitHub thread API), reply textarea grows on focus.
5. **Replace `CommentLineModal`** — removed entirely after 2.0 view zones land.

**Acceptance smoke list:**

- [ ] Submit review opens slide-in panel from the right (no modal anymore).
- [ ] Per-file `Mark viewed` toggle persists across app restarts.
- [ ] Resize the window under 1100 px → diff auto-switches to inline. Above → respects user pref (set in Settings).
- [ ] Thread widget reply works without leaving the diff. Resolve button hides the resolved thread.
- [ ] CommentLineModal no longer present.

## Sub-phase 2.5 — File tree advanced

**Branch:** `feat/etapa-2-5-file-tree`

**Scope:**

1. **Compact linear paths** — implement in `buildTree`, with `compactPaths: boolean` driven by settings (default ON). Separator: `.` if any segment matches `/^[a-z]+(\.[a-z]+)+$/` (Java-ish), else `/`.
2. **Flat mode** toggle in drawer header. Renders files as a flat list with `[name]   [dir]` columns; reuses search input to filter.
3. **File name search input** in the drawer header — filters both Tree and Flat views.
4. Setting toggle for compact linear paths in `Settings → Aparência`.

**Acceptance smoke list:**

- [ ] Open a Java repo PR (e.g. scorehub-api). Compact paths show `src/main/java/com.esparta.scorehub.<sub>` style joined nodes.
- [ ] Toggle compact off in Settings — tree expands back to per-segment nodes.
- [ ] Switch to Flat mode — file list displays with dir column. Search filters in both modes.
- [ ] Hovering a compact node shows full original path in tooltip.

## Sub-phase 2.6 — Settings refactor + repo add

**Branch:** `feat/etapa-2-6-settings-repos`

**Scope:**

1. **Settings refactor** into two-column layout: section list left, content right. Sections: Aparência, Repositórios, Filtros, Diff, Conta, Atalhos.
2. **Repo paste-URL parser** (`repo_input::parse(&str) -> Option<(String, String)>`) in Rust with unit tests. Supported forms listed in spec.
3. **Validate repo** Tauri command — calls `/repos/{owner}/{repo}`, surfaces error messages.
4. **Browse modal**:
   - New Tauri command: `list_my_repos(filters: { include_orgs, include_forks, include_archived })` → paginated fetch.
   - Modal with checkboxes, filter pills, search input, lazy pagination on scroll.
   - Save = sync (add new + remove unchecked).
5. **Settings → Diff**, **Settings → Conta**, **Settings → Atalhos** sections (read-only initially; Atalhos lists Ctrl+1/2/P/K/etc.).

**Acceptance smoke list:**

- [ ] Settings opens with sidebar nav. Switching sections changes content cleanly.
- [ ] Paste `https://github.com/esparta/scorehub-api/pull/42` into the URL input → resolves to `esparta/scorehub-api`, validates, saves.
- [ ] Paste bad URL → error "Repo não acessível com seu PAT" or "Formato inválido".
- [ ] "Procurar meus repos" opens modal with paginated list, filters work, checking + saving syncs config.
- [ ] Settings → Atalhos lists the keyboard shortcuts that are wired.

## Sub-phase 2.7 — Command palette + empty states

**Branch:** `feat/etapa-2-7-palette-polish`

**Scope:**

1. **Command palette** (`Ctrl+K`). Components: `<CommandPalette />`, `<CommandGroup />`, `<CommandItem />`. Fuzzy match using `fuse.js` or a simple custom matcher.
2. **Sources**:
   - PRs (open in any configured repo).
   - Files (within the current PR, if open).
   - Commands (Refresh PRs, Open Settings, Change theme, Submit review).
3. `>` prefix forces command-only mode.
4. **Empty states polish**: replace existing copy with `<EmptyState>` primitive for no-repos and no-PRs cases.
5. Optional: custom titlebar (Graphite-style) — only if time permits and user opts in at sub-phase start.

**Acceptance smoke list:**

- [ ] `Ctrl+K` opens palette. Typing fuzzy-matches PRs by title and number.
- [ ] With a PR open, typing also matches its files. Selecting a file selects it in the diff.
- [ ] `> refresh` lists the Refresh command.
- [ ] Empty states render the new polished cards with their CTAs.
- [ ] Final manual smoke checklist from the spec passes end-to-end.

## How to resume in a new Claude session

If context is lost or a new session starts:

1. Open `docs/superpowers/etapa-2-progress.md` — that's the source of truth for which sub-phase is active.
2. Open `docs/superpowers/specs/2026-05-23-etapa-2-design.md` — full design context.
3. Open this file — sub-phase scope + acceptance lists.
4. `git log --oneline --all | head -30` — see what's been built.
5. Continue the active sub-phase, or start the next one if the active one is marked `Done`.

A project memory entry (`pr-reviewer-etapa-2`) points to these files.
